# Unit 14 Mastery Quiz: Computer Vision

---

## I. Conceptual Questions

**1. CPU Resource Management:** Why is it considered standard engineering practice to call `visionPortal.stopStreaming()` once the autonomous vision decision has been made, rather than leaving the stream running for the entire autonomous period?

<details>
<summary>Show answer</summary>

A running camera stream consumes a significant share of the Control Hub's available CPU time. Frame capture, color space conversion, and pipeline execution all run continuously on a background thread and compete for the same processor cores that handle motor PID controllers, sensor polling, and the communication heartbeat with the Driver Station. Once the vision decision is made, continued streaming provides no additional value but continues to consume resources. Stopping the stream frees that CPU budget for the drivetrain and mechanism control loops, which improves their update frequency and makes autonomous movement more accurate and responsive.

</details>

---

**2. Timing and Pre-Match Readiness:** Why is the period between `init()` being pressed and `Start` being pressed -- specifically the `init_loop()` phase -- the optimal time to finalize the vision detection result rather than reading it in the first iteration of the main loop?

<details>
<summary>Show answer</summary>

The camera requires several frames to stabilize after opening, and the vision processor needs multiple iterations to produce a consistent, reliable result as it averages out random frame-to-frame noise. The `init_loop()` phase runs while the robot is stationary and the drive team is confirming the camera aim. By continuously displaying the current detection result during this phase, the team can visually verify that the correct location is identified before committing to the match. Reading the result in the very first iteration of the main loop gives the processor essentially zero warm-up time and risks capturing an unstable or incorrect detection.

</details>

---

**3. HSV vs. RGB Reliability:** Explain why a vision pipeline that compares Saturation values across zones is significantly more reliable for game element detection under varying field lighting than one that compares raw Red channel values against a fixed threshold.

<details>
<summary>Show answer</summary>

A fixed Red channel threshold is calibrated to a specific brightness level. Under brighter stadium lighting, the Red channel value for the same object increases, potentially exceeding any threshold. Under dimmer lighting, it decreases, potentially falling below it. The threshold requires manual recalibration for each venue. The Saturation channel measures color purity independent of overall brightness. A colored game element is highly saturated under bright light and under dim light. The grey field tiles surrounding it have low saturation under both conditions. Because the comparison is relative (which zone has higher saturation) rather than absolute (does this zone exceed value X), the detection remains robust across the full range of lighting conditions encountered at competition.

</details>

---

**4. The Null Metadata Guard:** A student writes a loop that accesses `detection.metadata.name` on every `AprilTagDetection` in the list without checking `detection.metadata != null` first. Under what specific circumstances will this cause a runtime crash, and how would you fix it?

<details>
<summary>Show answer</summary>

The `metadata` field is only populated when the detected tag's ID exists in the SDK's built-in field database of known FTC tags. If the camera detects a tag whose ID is not in that database -- for example, a custom tag on your practice field perimeter, a test tag on a whiteboard, or a tag from a previous season's database that has been updated -- the processor sets `metadata` to `null`. Accessing `.name` on a `null` reference immediately throws a `NullPointerException` and crashes the OpMode. The fix is a guard clause before any metadata access:

```java
if (detection.metadata != null) {
    String name = detection.metadata.name;
}
```

</details>

**5. Zone Boundaries and False Positives:** A team positions their `Rect` zones to cover only the spike mark areas and notices their detection is now more reliable at competition than it was in practice. Explain the structural reason why restricting the analysis region reduces false positive detections.

<details>
<summary>Show answer</summary>

A false positive occurs when the analysis algorithm finds evidence of the target in a region of the image that does not contain the target. Without zone restrictions, the full frame is analyzed, which includes field tile edges, alliance station walls, other robots, the scoring area, and anything else in the camera's field of view. Any of these could produce color or saturation values that match the criteria for a positive detection. By restricting the `Rect` zones to cover only the exact areas on the field where the game element could physically be placed, the algorithm can only report a positive result if high saturation is present in one of those three specific locations. Background objects that happen to match the detection criteria are structurally excluded because they fall outside all three zones.

</details>

## II. Debug the Code

The following code is intended to detect AprilTag ID 7 and display its forward distance to the Driver Station. **Identify and fix the 2 errors.**

:::note
This is an autonomous program. No gamepad input is used.
:::

```java
package org.firstinspires.ftc.teamcode;

import com.qualcomm.robotcore.eventloop.opmode.LinearOpMode;
import com.qualcomm.robotcore.eventloop.opmode.Autonomous;
import com.qualcomm.robotcore.hardware.WebcamName;
import org.firstinspires.ftc.vision.VisionPortal;
import org.firstinspires.ftc.vision.apriltag.AprilTagDetection;
import org.firstinspires.ftc.vision.apriltag.AprilTagProcessor;
import java.util.List;

@Autonomous(name="Vision_Debug")
public class VisionDebug extends LinearOpMode {

    private AprilTagProcessor aprilProcessor;
    private VisionPortal      portal;

    @Override
    public void runOpMode() {
        WebcamName webcam = hardwareMap.get(WebcamName.class, "Webcam 1");

        aprilProcessor = AprilTagProcessor.easyCreateWithDefaults();

        // Error 1: Incorrect portal creation
        portal = hardwareMap.get(VisionPortal.class, "Webcam 1");

        waitForStart();

        while (opModeIsActive()) {
            List<AprilTagDetection> detections = aprilProcessor.getDetections();

            // Error 2: Unsafe list access
            AprilTagDetection first = detections.get(0);
            telemetry.addData("Distance (in)", first.ftcPose.y);
            telemetry.update();
        }
    }
}
```

<details>
<summary>Show answers</summary>

**Error 1 -- Incorrect Portal Creation:**
A `VisionPortal` is not a hardware device registered in the Robot Controller configuration and cannot be retrieved using `hardwareMap.get()`. It must be instantiated using the SDK's static factory method, passing the already-retrieved webcam reference and the processor:

```java
portal = VisionPortal.easyCreateWithDefaults(webcam, aprilProcessor);
```

**Error 2 -- Unsafe List Access:**
Calling `detections.get(0)` without first verifying the list is non-empty will throw an `IndexOutOfBoundsException` whenever no tags are visible, which crashes the OpMode. The list must be checked before accessing any element:

```java
if (!detections.isEmpty()) {
    AprilTagDetection first = detections.get(0);
    if (first.metadata != null) {
        telemetry.addData("Distance (in)", first.ftcPose.y);
    }
}
```

</details>
