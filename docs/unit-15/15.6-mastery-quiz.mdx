# Unit 15 Mastery Quiz: Advanced Integration

---

## I. Conceptual Questions

**1. CPU Efficiency and Stream Selection:** Why is it standard engineering practice to call `limelight.pipelineSwitch(int index)` only once during initialization or a state transition rather than repeatedly calling it in every iteration of `loop()`?

<details>
<summary>Show answer</summary>

`pipelineSwitch()` changes which Limelight pipeline is active. If you repeatedly issue that command every loop cycle, you waste communication bandwidth and processing time telling the device to re-select a pipeline that is already active. Even if the Limelight treats repeated requests as a no-op, those extra calls still add unnecessary overhead and clutter the control loop. Calling `pipelineSwitch()` once during `init()` or only when the robot genuinely needs a different pipeline keeps the vision system stable and leaves the loop free to spend its time reading results and controlling hardware.

</details>

---

**2. Follower Consistency:** Within high-performance libraries like Pedro Pathing, why is the continuous execution of `follower.update()` imperative for maintaining the robot's geometric trajectory during unexpected field collisions?

<details>
<summary>Show answer</summary>

`follower.update()` is the step that reads the latest localization data, compares the robot's actual pose to the commanded path, and recomputes motor outputs to correct any deviation. After a collision or wheel slip event, the robot is no longer exactly where the path planner expected it to be. If `update()` is skipped, the Follower continues acting on stale position information and cannot apply the correction needed to recover the path. Running it every cycle ensures the robot continuously re-localizes and steers itself back onto the intended trajectory.

</details>

---

**3. Momentum Management:** Why do Bézier curves improve autonomous performance and mechanical longevity compared to traditional "stop-turn-drive" linear navigation methods?

<details>
<summary>Show answer</summary>

Bézier curves let the robot move through a continuous path instead of repeatedly stopping, rotating in place, and accelerating again. That produces smoother changes in heading and velocity, which reduces wheel scrub, slip, current spikes, and shock loads on drivetrains, gearboxes, and mechanism mounts. The robot also spends less time losing momentum between segments, so the autonomous routine is faster and more repeatable while placing less stress on the hardware.

</details>

---

**4. Error Correction via Sensor Fusion:** Why is Global Localization (re-localizing via AprilTags) necessary for long-duration autonomous routines, even when using high-accuracy encoders and a calibrated IMU?

<details>
<summary>Show answer</summary>

Encoders and an IMU are still relative sensors. They estimate motion from wheel rotation and rotational change, so small errors from wheel slip, imperfect calibration, floor seams, backlash, or minor impacts accumulate over time. AprilTag-based global localization gives the robot an absolute field reference that does not drift. By occasionally resetting the local pose estimate from a valid tag observation, the robot removes the accumulated error before it grows large enough to cause missed alignments later in the routine.

</details>

---

**5. Non-Blocking Logic:** Why is a state machine architecture using `enum` values and a `switch` block technically superior to sequential `LinearOpMode` logic when coordinating complex mechanism actions with simultaneous drivetrain movement?

<details>
<summary>Show answer</summary>

A state machine allows the loop to keep running continuously while different subsystems progress in parallel. Each iteration can call `follower.update()`, poll sensors, check timers, and decide whether the current phase is complete without blocking the rest of the robot. Sequential logic built around `sleep()` calls or long wait sections pauses the loop and prevents drivetrain corrections, telemetry updates, and mechanism coordination from happening at the same time. The state-machine approach scales better because each autonomous phase has a named state and a clear transition condition.

</details>

## II. Debug the Code

The following TeleOp is meant to use a Limelight 3A to track blue game elements with pipeline slot `0` and display the horizontal offset to the driver. **Identify and fix the one method-call error and one logic error.**

```java
package org.firstinspires.ftc.teamcode;

import com.qualcomm.robotcore.eventloop.opmode.OpMode;
import com.qualcomm.robotcore.eventloop.opmode.TeleOp;
import com.qualcomm.hardware.limelightvision.Limelight3A;
import com.qualcomm.hardware.limelightvision.LLResult;

@TeleOp(name="LimelightFix")
public class LimelightFix extends OpMode {

    Limelight3A visionDevice;

    @Override
    public void init() {
        visionDevice = hardwareMap.get(Limelight3A.class, "limelight");
        visionDevice.start();
    }

    @Override
    public void loop() {
        // Error 1: Wrong pipeline argument type
        visionDevice.pipelineSwitch("blue_tracking");

        LLResult result = visionDevice.getLatestResult();

        // Error 2: Missing result validation before data access
        double tx = result.getTx();

        telemetry.addData("Horizontal Offset", "%.2f", tx);
        telemetry.update();
    }
}
```

<details>
<summary>Show answers</summary>

**Error 1 -- Wrong pipeline argument type:**
`pipelineSwitch()` expects an integer slot index, not a `String`. The correct call is:

```java
visionDevice.pipelineSwitch(0);
```

**Error 2 -- Missing result validation:**
`getLatestResult()` can return `null` before the Limelight finishes processing its first frame, and a non-null result may still be invalid if no target is visible. Accessing `result.getTx()` without checking both conditions risks a runtime crash or invalid telemetry. The correct pattern is:

```java
LLResult result = visionDevice.getLatestResult();

if (result != null && result.isValid()) {
    double tx = result.getTx();
    telemetry.addData("Horizontal Offset", "%.2f", tx);
} else {
    telemetry.addData("Horizontal Offset", "No valid target");
}
telemetry.update();
```

</details>
