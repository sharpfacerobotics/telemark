# Lesson 15.4: Global Field Localization with AprilTag and Limelight Fusion

---

## Why Odometry Alone Is Not Enough for a Two-Minute Match

Dead-wheel odometry is highly accurate in the short term. Over 5 to 10 seconds and a few feet of travel, the position estimate from three dead-wheel encoders is typically within a centimeter of the robot's true position. Over two minutes and the full scoring cycle of a competitive autonomous routine -- driving from the starting tile, scoring, navigating to the sample area, picking up, returning, and repeating -- small errors integrate into position drift that can accumulate to several inches or more.

For a robot that needs to align with a centimeter-scale target like a specific basket or a backdrop slot, a 3-inch drift at the end of the autonomous period is the difference between scoring and missing. The fix is to periodically correct the odometry estimate using an absolute reference rather than allowing the error to compound uninterrupted.

---

## How Limelight AprilTag Fusion Works

The FTC field has AprilTags mounted at known, fixed coordinates relative to the field origin. When the Limelight detects an AprilTag, it computes the robot's position and orientation in the global field frame using the known tag position and the tag's apparent pose in the camera image. This global position is available through `result.getBotpose()`, which returns a `Pose3D` object containing the robot's X, Y, Z, and angular position relative to the field origin.

Crucially, this position is global. It does not drift. No matter how many times the robot has turned, bumped a wall, or driven over a rough tile seam, a single valid Limelight AprilTag observation corrects the position estimate back to reality. The workflow is:

1. The robot drives using Pedro Pathing with dead-wheel odometry.
2. Whenever a valid Limelight AprilTag detection is available, extract the `Pose3D`.
3. Convert the `Pose3D` to Pedro Pathing's `Pose` format and call `follower.setPose()` to reset the localizer to the corrected position.
4. Pedro Pathing immediately resumes path tracking from the corrected position, discarding the accumulated drift.

---

## Extracting Position from Pose3D

The `Pose3D` object returned by `getBotpose()` has two sub-objects: `getPosition()` and `getOrientation()`. Position returns an `Position3D` with `x`, `y`, and `z` in meters (not inches). Orientation returns angle data in the `YawPitchRollAngles` format introduced in Unit 12. When feeding this data to Pedro Pathing -- which uses inches -- unit conversion is required.

```java
Pose3D botPose = result.getBotpose();

double xInches      = botPose.getPosition().x * 39.3701;
double yInches      = botPose.getPosition().y * 39.3701;
double headingRads  = botPose.getOrientation().getYaw(AngleUnit.RADIANS);

follower.setPose(new Pose(xInches, yInches, headingRads));
```

The correction should only be applied when the Limelight result is valid and the tag detection quality is high enough to be trustworthy. A single noisy frame can introduce a large correction error. Checking `result.isValid()` is mandatory, and some implementations also filter on tag detection confidence or require the same position to be consistent across multiple consecutive frames before applying it.

## Annotated Code

:::note
This is an autonomous program. No gamepad input is used.
:::

```java
package org.firstinspires.ftc.teamcode;

import com.qualcomm.robotcore.eventloop.opmode.LinearOpMode;
import com.qualcomm.robotcore.eventloop.opmode.Autonomous;
import com.qualcomm.hardware.limelightvision.Limelight3A;
import com.qualcomm.hardware.limelightvision.LLResult;
import com.pedropathing.follower.Follower;
import com.pedropathing.localization.Pose;
import org.firstinspires.ftc.robotcore.external.navigation.AngleUnit;
import org.firstinspires.ftc.robotcore.external.navigation.Pose3D;

@Autonomous(name="Limelight_Fusion_Demo")
public class LimelightFusionDemo extends LinearOpMode {

    private Follower    follower;
    private Limelight3A limelight;

    // AprilTag detection pipeline on the Limelight
    private static final int APRILTAG_PIPELINE = 0;

    // Meters to inches conversion factor
    private static final double M_TO_IN = 39.3701;

    @Override
    public void runOpMode() {
        follower  = new Follower(hardwareMap);
        limelight = hardwareMap.get(Limelight3A.class, "limelight");

        follower.setStartingPose(new Pose(12, 12, 0));

        limelight.pipelineSwitch(APRILTAG_PIPELINE);
        limelight.start();

        waitForStart();

        // Path following loop with integrated Limelight correction
        while (opModeIsActive() && follower.isBusy()) {
            // Always update the Follower first
            follower.update();

            // Attempt a position correction from the Limelight
            applyLimelightCorrection();

            telemetry.addData("X",    "%.2f in", follower.getPose().getX());
            telemetry.addData("Y",    "%.2f in", follower.getPose().getY());
            telemetry.update();
        }

        limelight.stop();
    }

    /**
     * Reads the latest Limelight AprilTag result and, if valid,
     * resets the Pedro Pathing localizer to the globally-corrected position.
     */
    private void applyLimelightCorrection() {
        LLResult result = limelight.getLatestResult();

        if (result == null || !result.isValid()) {
            return; // No valid observation this cycle -- keep current estimate
        }

        Pose3D botPose = result.getBotpose();

        if (botPose == null) {
            return; // AprilTag pipeline did not see a tag this frame
        }

        double xInches     = botPose.getPosition().x * M_TO_IN;
        double yInches     = botPose.getPosition().y * M_TO_IN;
        double headingRads = botPose.getOrientation().getYaw(AngleUnit.RADIANS);

        // Overwrite the dead-wheel estimate with the globally-corrected pose
        follower.setPose(new Pose(xInches, yInches, headingRads));

        telemetry.addData("Correction Applied", "x=%.1f y=%.1f", xInches, yInches);
    }
}
```
